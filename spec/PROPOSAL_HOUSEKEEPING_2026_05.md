# Housekeeping: UI File Decomposition + Test Gaps (May 2026)

> **Status: PROPOSED** — drafted 2026-05-04 after a from-scratch audit. Supersedes the
> April refactor sprint that was paused before it could land all planned splits.

A focused, phased cleanup of the largest and messiest source files in the project, plus a
short list of structural test gaps. Emphasis is on the UI tier (CSS + JS modules) which has
grown well past the thresholds we used to defer earlier in 2026; the Java tier is mostly
healthy and only two services warrant action.

This is **not** a re-attempt of the April "big refactor." That sprint's value was the
state-factory groundwork in `title-browse.js` / `actress-browse.js` and the dashboard-renderer
extraction; that work shipped and we keep it. This proposal is narrower: split files that grew
during the post-April feature work, and pin the few service files that have no test at all.

---

## 1. Motivation

Two pressures justify acting now rather than continuing to defer:

- **Navigation pain.** `css/av-tools.css` has reached 7,505 lines and contains styles for nine
  unrelated features. Searching it is no longer reliable; section headers are the only
  structure. `utilities-javdb-discovery.js` (2,604 lines) is the largest JS module by a wide
  margin and cuts cleanly along its own subtab boundaries.
- **Coverage uncertainty.** The on-disk JaCoCo report is stale (generated by a targeted
  `--tests` run on 2026-05-04); per-class coverage in memory dates to 2026-04-20. Several
  high-traffic route files have no test file matching their name. Until we add baseline tests,
  any further refactor is ungrounded.

Non-goals:

- No semantic changes. Every split must be byte-equivalent in rendered output and behavior.
- No re-architecture of the UI shell, no framework introduction, no Spark template engine.
- No attempt to reach a coverage % target. The test work is gap-filling, not blanket sweeping.

## 3. Phased plan

### Phase 1 — Test pinning (do this first)

**Goal:** every file we plan to touch in Phases 2-4 has a regression baseline before it moves.

1. Add `UtilitiesRoutesTest` covering the route handlers in
   `web/routes/UtilitiesRoutes.java` (627L). Each top-level endpoint gets one happy-path and
   one error-path assertion. Use the existing `WebRouteTestHarness` pattern from
   `DraftRoutesTest`.
2. Add `JavdbDiscoveryRoutesTest` for `web/routes/JavdbDiscoveryRoutes.java` (269L). Same
   pattern.
3. Add a Playwright safety-net suite for the JavDB Discovery tool: subtab switching
   (enrich / titles / collections / queue / review), filter input behavior, enqueue button
   state. Mirrors `UiTitleBrowseTest` in shape; aim for 6-10 tests.
4. Add `UtilitiesJavdbDiscoveryStateTest` if any pure-logic helpers exist post-split (deferred
   decision — write only if §4.2 turns up extractable functions).

**Exit criterion:** `./gradlew test --tests 'Utilities*' --rerun` and the new Playwright tests
all pass before any structural code change is pushed.

### Phase 2 — `av-tools.css` split

7,505 lines → ~8 files. Inferred boundaries from current section headers:

| New file | Approximate source range | Notes |
|---|---|---|
| `css/av-tools.css` (slimmed) | AV landing, AV index, AV cards, AV detail | ~1,500L, original-purpose only |
| `css/logs.css` | Logs viewer subview | ~90L |
| `css/aliases.css` | Aliases two-pane editor | ~350L |
| `css/duplicates.css` | Duplicates filter + tool | ~130L |
| `css/enrichment-review.css` | Picker / candidate cards / footer | ~600L |
| `css/no-match-triage.css` | nmt-* classes | ~260L |

| `css/title-peek.css` | Title peek modal + cover lightbox | ~200L |
| `css/enrichment-detail-modal.css` | Enrichment detail modal | ~150L |

Procedure:
1. For each new file: `mv` the section, no edits, no rewrites.
2. Add `<link rel="stylesheet">` for each new file in `index.html`, ordered after `base.css`
   and before view-specific files, preserving original cascade order.
3. Diff rendered output by loading the app pre/post and visually checking each tool view, plus
   a Playwright screenshot diff if practical.

**Exit criterion:** zero visual regressions across all utilities tools and AV detail pages.

### Phase 3 — `utilities-javdb-discovery.js` split

2,604L → directory layout:

```
modules/utilities-javdb-discovery/
  index.js          # state factory, tab dispatch, lifecycle hooks (~400L)
  enrich.js         # actress alpha bar, filter, sort, enrich flow (~600L)
  titles.js         # titles tab body + filter + autocomplete (~500L)
  collections.js    # collections tab (~400L)
  queue.js          # queue tab (~300L)
  review.js         # enrichment review subview (~400L)
```

Procedure: identical to title-browse refactor in April — single shared `state` object passed
to each module's exported handlers; DOM handles fetched lazily inside each module. Existing
import path (`modules/utilities-javdb-discovery.js`) becomes a re-export shim or is updated at
the call site (one place: `nav.js`).

**Exit criterion:** Playwright suite from §3.1 still passes; tab switching and queue ticker
remain identical.

### Phase 5 — Java service decomposition (lowest priority, may defer to its own session)

`ActressBrowseService.java` (1,194L) and `TitleBrowseService.java` (741L) build dashboard
payloads inline. Extract the dashboard-builder methods into sibling classes
(`ActressDashboardBuilder`, `TitleDashboardBuilder`) — analogous to the
`dashboard-renderers.js` extraction on the frontend. Each builder gets a focused unit test.

**Exit criterion:** existing service tests still green; new builder classes have direct unit
tests.

## 4. Risks and mitigations

| Risk | Mitigation |
|---|---|
| CSS cascade order changes during split, breaking selectors that relied on later-file precedence | Preserve original `<link>` order; verify by diffing computed styles on a representative page pre/post |
| JS split introduces import cycles between sibling modules | Pass shared `state` object explicitly; sibling modules never import each other, only `index.js` |
| Playwright tests flake under CI even when behavior is unchanged | Add tests in Phase 1 against the *current* code first, observe stability over a day, then proceed |
| `UtilitiesRoutes` test reveals undocumented behavior we then have to preserve | Acceptable — documenting current behavior is part of the value |
| Refactor takes longer than estimated and stalls feature work | Each phase is independently mergeable; pause between phases is fine |

## 5. Test gaps independent of the refactor

These are gaps worth filling regardless of whether Phases 2-5 happen:

1. `web/routes/UtilitiesRoutes` — 627L, no test file. **Highest priority.**
2. `web/routes/JavdbDiscoveryRoutes` — 269L, no test file.
3. `web/routes/UnsortedEditorRoutes` (212L), `TitleTagEditorRoutes` (163L),
   `CustomAvatarRoutes` (125L), `VideoRoutes` (180L). No tests; routes for tier-1 features.
4. `media/ThumbnailService` — 323L, only `ThumbnailServiceBlockingTest` exists; pure-logic
   parts (path resolution, frame-time selection, cache key construction) are untested.
5. `utilities/task/organize/PrepBaseTask` (169L) and `OrganizeBaseTask` (139L). Verify
   subclass tests exercise the base logic before deciding whether to add direct tests.

Translation, trash, sync, and enrichment packages have substantial test suites already — no
gaps worth filing here.
